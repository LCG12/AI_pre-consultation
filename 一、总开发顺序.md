# 一、总开发顺序

```
1. 明确一期范围：只做发热 / 咳嗽 / 咽痛2. 定义 Preconsult State3. 定义 fever_cough_v1 问诊路径4. 定义红旗规则、黄色规则、科室规则5. 定义追问模板、患者结果模板、医生摘要模板6. 搭建 FastAPI 后端7. 搭建 SQLite 本地存储8. 实现会话接口和消息接口9. 实现红旗关键词预检10. 实现规则引擎 RiskEngine / TriageEngine11. 实现 Slot Planner12. 实现模板追问13. 接入 LLM 字段抽取14. 跑通完整 handle_message 流程15. 再接入 LangGraph 工作流16. 实现患者预分诊结果17. 实现医生摘要异步生成18. 实现安全拦截19. 建立测试病例库20. Docker 部署21. 机器人联调22. 再做知识库增强23. 后期再扩展其他症状路径
```

---

# 二、第一阶段：先做最小后端闭环

第一阶段不要做复杂功能，先做一个路径：

```
fever_cough_v1：发热 / 咳嗽 / 咽痛
```

先跑通这个闭环：

```
创建会话  ↓机器人传入患者文本  ↓后端读取 State  ↓抽取字段  ↓更新 State  ↓判断红旗风险  ↓判断是否继续追问  ↓返回下一句问题或最终预分诊结果
```

这一阶段先不做：

```
复杂 RAG模型微调PostgreSQL 正式库复杂后台管理多病种HIS 对接医生端复杂页面
```

---

# 三、第 1 步：定义 Preconsult State

这是第一步，也是最重要的基础。

你们先定义统一状态对象：

```
{  "session_id": "s_10001",  "status": "in_progress",  "source": "robot",  "robot_context": {    "robot_id": "robot_001",    "location": "门诊大厅"  },  "path_id": "fever_cough_v1",  "chief_complaint": {    "raw_text": "",    "main_symptoms": [],    "duration_days": null  },  "patient_basic_info": {    "age": null,    "gender": null,    "pregnancy_status": null  },  "slots": {    "fever": {      "duration_days": null,      "max_temperature_c": null,      "current_temperature_c": null    },    "cough": {      "has_cough": null,      "cough_type": null,      "sputum": null    },    "associated_symptoms": {      "sore_throat": null,      "fatigue": null,      "headache": null,      "diarrhea": null,      "rash": null    },    "red_flags": {      "shortness_of_breath": null,      "chest_pain": null,      "hemoptysis": null,      "confusion": null,      "seizure": null,      "cyanosis": null    },    "past_history": {},    "allergy_history": {},    "medication_history": {}  },  "risk": {    "current_level": "unknown",    "rule_hits": []  },  "dialogue": {    "turn_count": 0,    "max_turns": 5,    "missing_required_slots": []  }}
```

# 五、第 3 步：定义规则

先写规则配置，不急着写复杂代码。

## 1. 红旗规则

```
configs/rules/red_flags.json
```

包括：

```
呼吸困难胸痛咳血意识异常抽搐嘴唇发紫
```

示例：

```
{  "rule_id": "RED_DYSPNEA",  "name": "呼吸困难",  "condition": {    "field": "slots.red_flags.shortness_of_breath",    "operator": "equals",    "value": true  },  "risk_level": "red",  "reason": "存在呼吸困难",  "should_stop_dialogue": true}
```

## 2. 黄色规则

```
configs/rules/yellow_flags.json
```

示例：

```
{  "rule_id": "YELLOW_FEVER_3_DAYS",  "name": "发热持续3天及以上",  "condition": {    "field": "slots.fever.duration_days",    "operator": ">=",    "value": 3  },  "risk_level": "yellow",  "reason": "发热持续3天及以上"}
```

## 3. 科室规则

```
configs/rules/department_rules.json
```

示例：

```
{  "rule_id": "DEPT_FEVER_COUGH",  "condition": {    "field": "chief_complaint.main_symptoms",    "operator": "contains_any",    "value": ["fever", "cough", "sore_throat"]  },  "departments": ["呼吸内科", "全科"],  "reason": "主诉为发热、咳嗽、咽痛等呼吸道相关症状"}
```

# 八、第 6 步：先实现不用 LLM 的规则版流程

在接入 LLM 前，先把规则版跑通。

实现这些模块：

```

```

处理流程：

```
机器人传入文本  ↓读取 State  ↓红旗关键词预检  ↓规则判断  ↓Slot Planner 找缺失字段  ↓TemplateTool 返回下一句追问
```

> 不用 LLM，也能跑通会话、追问、红旗急诊提示、最终预分诊的骨架。

# 九、第 7 步：实现红旗关键词预检

红旗预检必须早做。

配置：

```
{  "shortness_of_breath": ["喘不上气", "呼吸困难", "憋气", "气短", "喘不过来"],  "chest_pain": ["胸痛", "胸口疼", "胸口压着疼", "胸闷明显"],  "hemoptysis": ["咳血", "痰里有血", "血丝痰"],  "confusion": ["意识不清", "叫不醒", "反应差", "神志不清"],  "seizure": ["抽搐", "惊厥"],  "cyanosis": ["嘴唇发紫", "脸色发紫"]}
```

患者说：

```
胸口疼，喘不上气
```

后端直接返回：

```
{  "status": "emergency",  "risk_level": "red",  "reply": "您提到胸痛和呼吸困难，这属于需要尽快处理的风险信号。请立即联系现场医护人员，或前往急诊。",  "recommended_departments": ["急诊"],  "should_stop_dialogue": true}
```

这条路径不调用 LLM。

# 十、第 8 步：接入 LLM 字段抽取

等规则版流程跑通后，再接模型。

模型只负责：

```
主诉识别字段抽取症状归一化否定表达识别不确定表达识别证据片段提取
```

模型不负责：

```
风险分层科室推荐诊断开药治疗建议
```

字段抽取输出格式固定：

```
{  "chief_complaint": {    "raw_text": "",    "main_symptoms": [],    "duration_days": null,    "path_id": ""  },  "extracted_slots": {},  "uncertain_fields": [],  "raw_evidence": []}
```

示例：

```
我发烧三天，最高38度8，没有胸痛，也没有喘。
```

期望抽取：

```
{  "chief_complaint": {    "raw_text": "我发烧三天，最高38度8，没有胸痛，也没有喘。",    "main_symptoms": ["fever"],    "duration_days": 3,    "path_id": "fever_cough_v1"  },  "extracted_slots": {    "slots.fever.duration_days": 3,    "slots.fever.max_temperature_c": 38.8,    "slots.red_flags.chest_pain": false,    "slots.red_flags.shortness_of_breath": false  },  "uncertain_fields": [],  "raw_evidence": [    {      "field": "slots.fever.duration_days",      "text": "发烧三天"    },    {      "field": "slots.fever.max_temperature_c",      "text": "最高38度8"    },    {      "field": "slots.red_flags.chest_pain",      "text": "没有胸痛"    },    {      "field": "slots.red_flags.shortness_of_breath",      "text": "没有喘"    }  ]}
```

# 十一、第 9 步：跑通完整 handle_message

完整流程：

```
读取 State  ↓保存 patient message  ↓红旗关键词预检  ↓LLM 字段抽取  ↓合并 State  ↓风险规则判断  ↓Slot Planner  ↓TemplateTool 生成追问  ↓保存 assistant message  ↓保存 State  ↓返回机器人
```

这一步完成后，你们已经有了可用的核心后端。

---

# 十二、第 10 步：再接入 LangGraph

不要一开始就用 LangGraph 写复杂流程。

先用普通 Python 函数跑通，再封装成 LangGraph 节点。

节点：

```
precheck_red_flagsextract_and_update_staterisk_gatekeeperplan_or_triageresponse_builder
```

图结构：

```
START  ↓precheck_red_flags  ├── emergency_response  └── extract_and_update_state          ↓     risk_gatekeeper          ├── emergency_response          └── plan_or_triage                  ├── inquiry_response                  └── triage_response
```

LangGraph 只做编排，不做医学判断。

# 最终建议

你们最合理的开发顺序是：

```
1. State2. 路径配置3. 规则配置4. 追问模板5. FastAPI6. SQLite7. 无 LLM 规则版流程8. LLM 字段抽取9. 完整 handle_message10. LangGraph11. 患者预分诊结果12. 医生摘要13. 安全拦截14. 测试病例15. Docker 部署16. 机器人联调
```

最关键的一句话：

> **先把发热/咳嗽路径的规则版后端跑通，再接 LLM 做字段抽取，最后再用 LangGraph 工程化工作流。**
